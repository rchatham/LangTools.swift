### Still Open

#### 1 `@retroactive` missing — **high priority**

`Sources/LangTools/LangTools+Speech.swift` still reads:

```swift
extension Data: LangToolsAudioResponse { ... }
extension String: LangToolsTranscriptionResponse { ... }
```

Neither has `@retroactive`. The bot commit listed this as fixed but the annotation is absent from the current file. Swift 5.7+ warns; Swift 6 strict builds error.

```swift
// Required fix:
extension Data: @retroactive LangToolsAudioResponse { ... }
extension String: @retroactive LangToolsTranscriptionResponse { ... }
```

#### 2 `startRecognition() throws` silently swallows errors — **high priority**

`Sources/WhisperKitLangTools/WhisperKitSpeechRecognitionProvider.swift`:

```swift
public func startRecognition() throws {
    Task {
        try await startStreamingTranscription { ... }   // error silently dropped
    }
}
```

The function signature promises `throws`, but no error can ever reach the caller — it is discarded inside the unstructured `Task {}`. Callers wrap this in `try` expecting failure feedback (e.g. model not loaded, microphone unavailable), but they will never receive it. Either propagate the error via `async throws` and make the protocol method `async throws`, or surface errors through `eventHandler` and remove the misleading `throws`.

#### 3 `stopRecognition` / `finalizeRecognition` write actor-isolated state after `await` — **medium priority**

```swift
public func stopRecognition(finalizePending: Bool, clearTranscript: Bool) {
    Task {
        let text = await stopStreamingTranscription()   // suspends here
        currentTranscript = text                        // @MainActor write after resume
        eventHandler?(.finalTranscription(text))
    }
}
```

After the `await` suspension, the task resumes on whatever executor `stopStreamingTranscription` returns from. If that function is not `@MainActor`, the writes to `currentTranscript` and the `eventHandler` call are not guaranteed to be on the main actor. Swift 6 strict concurrency will diagnose this. The minimal safe fix is:

```swift
Task { @MainActor in
    let text = await stopStreamingTranscription()
    currentTranscript = text
    eventHandler?(.finalTranscription(text))
}
```

or wrap the mutations in `await MainActor.run { ... }`. Same pattern applies to `finalizeRecognition`.

#### 4 `WhisperKitSTTProvider.configure` still writes to global singleton — **medium priority**

```swift
public func configure(languageIdentifier: String) {
    ToolSettings.shared.sttLanguage = .init(rawValue: languageIdentifier) ?? ToolSettings.shared.sttLanguage
    provider.configure(languageIdentifier: languageIdentifier)
}
```

All other providers store language locally; only this one mutates a shared singleton as a side-effect of `configure()`. This couples the example-app adapter to `ToolSettings`, makes the provider non-testable in isolation, and risks cross-provider interference if two providers are configured in the same session.

#### 5 `print` statements — **low priority**

- `AppleSpeechSTTProvider.updateLocaleFromSettings()` — `print("[AppleSpeech] Changed locale to: \(locale.identifier)")` is a development trace, not a user-facing log. Remove before merge.
- `NetworkClient.playAudio` — the `catch` block calls `print(error.localizedDescription)` and silently discards the error. The error should be rethrown or surfaced to the UI layer; swallowing it means TTS failures are invisible to callers.

### Minor

- `LangToolsSpeechTranscriptionRequest` doc comment says "at least one audio source should be non-nil" but this is advisory only. A `func validate() throws` protocol requirement — or even a simple precondition in the default extension — would catch misconfigured conformances at the abstraction boundary rather than deep inside provider implementations.
- `AppleSpeechSTTProvider.startRecognition()` passes `onPartialResult: { _ in }` / `onFinalResult: { _ in }` closures. If these callbacks are intentionally ignored in favour of `eventHandler`, a comment at the call site would prevent a future reader from assuming they are wired up.

### Summary

Items #1–#3 are the blockers before merge. The `@retroactive` fix is a one-liner. Issues #2 and #3 involve the same `WhisperKitSpeechRecognitionProvider` — fixing the actor context (#3) and rethinking the error surface (#2) can likely be done in a single pass on that file.
